# EAI日志解析器部署说明

## 更新日期: 2026-01-12

## 修改内容

### 问题描述
之前的版本使用工单号前缀(WONO)来推断产线信息，导致产线识别不准确。

### 解决方案
产线(LINE)字段应该直接从EAI日志的触发器数据中提取，而不是推断。

触发器数据格式示例：
```
[INFO][2025-12-08 14:41:33.697][flow_log.go:62 EventLog][ source=FLOW_DP-EPS\IPA MES报工接口]
>>>>>>>>>>>>>>>>>>db trigger get data:[{"CNT":"16","LINE":"DP EPS1","PACKID":"20251208E2700122","PARTNO":"H25.910.002","WONO":"EPS25120304"}]
```

### 代码修改

1. **新增 `TriggerData` 数据类** - 存储从触发器中提取的数据
   - LINE: 产线
   - PACKID: 批次号
   - WONO: 工单号
   - CNT: 数量
   - PARTNO: 料号

2. **新增触发器解析正则** `TRIGGER_DATA_PATTERN` - 匹配 `db trigger get data:[...]`

3. **新增 `_handle_trigger_data()` 方法** - 解析并缓存触发器数据

4. **修改 `_handle_request()` 方法** - 从缓存的触发器数据中获取LINE

5. **修改 `_handle_response()` 方法** - 成功/失败后都清空触发器缓存

6. **删除 `WONO_LINE_MAPPING` 和 `LINE_MAPPING`** - 不再使用推断逻辑

7. **删除 `extract_line_from_wono()` 方法** - 不再需要

### 处理流程

```
1. 解析到 `db trigger get data:[...]`
   -> 提取JSON并缓存TriggerData（特别是LINE字段）

2. 解析到 `kingdee request json:...`
   -> 关联之前缓存的TriggerData，获取LINE
   -> 缓存当前请求

3. 解析到 `kingdee response json:...`（成功）
   -> 使用触发器中的LINE写入数据库
   -> 清空触发器和请求缓存
```

## 测试验证

可以使用以下Python代码验证修改是否正确：

```python
from log_parser import LogParser

parser = LogParser()

# 模拟触发器数据
trigger_line = r'[INFO][2025-12-08 14:41:33.697]... db trigger get data:[{"CNT":"16","LINE":"DP EPS1","PACKID":"20251208E2700122","PARTNO":"H25.910.002","WONO":"EPS25120304"}]'
parser.parse_line(trigger_line)
print(f"触发器LINE: {parser._current_trigger.line}")  # 应输出: DP EPS1

# 模拟请求
request_line = r'[INFO]... kingdee request json:{"data":"{\"FMoBillNo\":\"EPS25120304\"}","url":"..."}'
parser.parse_line(request_line)
print(f"请求中的产线: {parser._current_request[3]}")  # 应输出: DP EPS1
```
