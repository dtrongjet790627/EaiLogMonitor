## 二、日志解析规则

### 2.1 EAI日志处理流程

```
db trigger get data:[{...}]     ← 1. 触发器数据（包含LINE、PACKID等）
        ↓
kingdee request json: {...}     ← 2. ERP请求
        ↓
kingdee response json: {...}    ← 3. ERP响应（成功/失败）
        或
run error: call lua error:...   ← 3. Lua执行错误
```

### 2.2 字段提取规则

#### 触发器数据 (db trigger get data)
```
格式: db trigger get data:[{"CNT":"16","LINE":"DP EPS1","PACKID":"20251208E2700122","PARTNO":"H25.910.002","WONO":"EPS25120304"}]

提取字段:
- LINE: 产线名称（直接使用，不推断）
- PACKID: 批次号
- WONO: 工单号
- CNT: 数量
- PARTNO: 料号
```

#### 成功响应 (IsSuccess: true)
```json
{"Result":{"ResponseStatus":{"IsSuccess":true,"SuccessEntitys":[{"Number":"SCHB00071247"}]}}}

提取: Number → 汇报单号(SCHB_NUMBER)
```

#### 失败响应 (IsSuccess: false)
```json
{"Result":{"ResponseStatus":{"IsSuccess":false,"Errors":[{"Message":"错误描述"}]}}}

提取: Errors[].Message → 错误信息(ERROR_MESSAGE)
```

#### Lua执行错误 (run error)
```
run error: call lua error:... {"errorMsg":"...\"Message\":\"具体错误\"..."}

提取: 嵌套JSON中的Message字段
```

### 2.3 错误信息清理规则

提取Message后需清理：
1. 去除转义序列: `\r\n`, `\n`, `\r`
2. 截断到中文分号 `；`（分号前是核心信息）
3. 截断到反斜杠 `\`（后面通常是JSON结构）

示例：
- 原始: `明细第1行的物料编码值为:【D24.204.232】不存在；\r\n\",\"DIndex\":0}]...`
- 清理后: `明细第1行的物料编码值为:【D24.204.232】不存在`

## 三、数据库表结构

### 3.1 表定义 (ACC_ERP_REPT_SUCC)

```sql
CREATE TABLE ACC_ERP_REPT_SUCC (
    ID NUMBER PRIMARY KEY,
    WONO VARCHAR2(50),              -- 工单号
    PACKID VARCHAR2(50),            -- 批次号
    PARTNO VARCHAR2(50),            -- 料号
    CNT NUMBER,                     -- 数量
    LINE VARCHAR2(50),              -- 产线
    SCHB_NUMBER VARCHAR2(50),       -- 汇报单号（成功时有值）
    SOURCE_BILL_NO VARCHAR2(50),    -- 源单号
    REPORT_TIME DATE,               -- 报工时间
    CREATETIME DATE DEFAULT SYSDATE,-- 记录创建时间
    ERROR_MESSAGE VARCHAR2(2000),   -- 错误信息（失败时有值）
    IS_SUCCESS NUMBER(1) DEFAULT 1  -- 成功标志（1=成功，0=失败）
);
```

### 3.2 字段说明

| 字段 | 成功时 | 失败时 |
|------|--------|--------|
| SCHB_NUMBER | 有值 | 空 |
| ERROR_MESSAGE | 空 | 有值 |
| IS_SUCCESS | 1 | 0 |

## 四、常见问题与处理

### 4.1 产线(LINE)字段为空

**原因**: 触发器数据解析失败或日志格式变化

**排查步骤**:
1. 检查原始日志格式: `ssh -p 2200 root@172.17.10.163 "tail -100 /var/eai/logs/xxx.log"`
2. 确认 `db trigger get data:` 后的JSON格式
3. 检查 `log_parser.py` 中的 `TRIGGER_DATA_PATTERN` 正则

**解决方案**: 修改正则匹配规则适配新格式

### 4.2 错误信息显示不完整或过多

**原因**: 正则匹配范围不当

**排查步骤**:
1. 获取原始日志中的 `run error` 行
2. 测试正则匹配结果
3. 调整清理逻辑

**解决方案**: 修改 `app.py` 中的错误信息清理代码

### 4.3 服务无法连接EAI服务器

**原因**: SSH连接问题

**排查步骤**:
1. 测试SSH: `ssh -p 2200 root@172.17.10.163 "echo OK"`
2. 检查密钥配置
3. 查看服务日志

**解决方案**: 重新配置SSH密钥或检查网络

## 六、去重机制

### 6.1 成功记录去重
- 基于 `SCHB_NUMBER`（汇报单号）去重
- 使用内存缓存提高效率
- 服务启动时加载已有工单号到缓存

### 6.2 失败记录去重（2026-01-12新增）
- 基于 `WONO + PACKID + ERROR_MESSAGE` 组合去重
- 解决EAI重试产生大量重复错误日志的问题
- 每次插入前查询数据库判断是否已存在相同记录

**应用场景**：
EAI接口在工单操作失败时会自动重试，每次重试都会生成新的日志，但错误原因相同。例如：
- 工单 `MID-226010703`
- 批次 `20250110`
- 错误 `明细第1行的物料编码值为:【D24.204.232】不存在`

重试24次会产生24条相同的日志，启用去重后只保留第一条。

**技术实现**（db_handler.py）：
```python
def is_error_duplicate(self, wono, packid, error_message):
    """检查是否存在相同的失败记录"""
    cursor.execute("""
        SELECT COUNT(*) FROM ACC_ERP_REPORT_SUCCESS
        WHERE WONO = :wono
        AND NVL(PACKID, '') = NVL(:packid, '')
        AND ERROR_MESSAGE = :error_message
        AND IS_SUCCESS = 0
    """, {...})
```
